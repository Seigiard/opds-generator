17 ноя 2026Chrome 158 удаляет XSLT
Форсирующий фактор — меняет постановку задачи

Chrome удаляет XSLT из браузера: предупреждения с Chrome 143 (дек 2025), полное отключение в стабильном Chrome 158 — 17 ноября 2026 (Origin-Trial-исключения до авг 2027). Firefox и WebKit поддержали удаление. Официальный WASM-полифилл Google воспроизводит только явные вызовы XSLTProcessor из JS — он не умеет перехватывать <?xml-stylesheet?>-автотрансформацию, то есть ровно тот механизм, на котором работает текущий viewer. Вопрос «как синхронизировать разработку с layout.xsl» стал вопросом «что заменит layout.xsl» — синхронизация решается выбором замены. Источники: Chrome for Developers, Chrome Platform Status.

Grounding Context

Codebase Context. Продакшн (opds-generator): человеческий UI — браузерный XSLT 1.0; каждый feed.xml получает PI <?xml-stylesheet href="/static/layout.xsl"?> (инжектится в src/server.ts и src/effect/handlers/folder-meta-sync.ts). static/layout.xsl (238 строк) грузит style.css, рукописный main.js (Gridnav-клавонавигация) и два CDN-скрипта с unpkg (tabbable, focus-trap). Дев-плейграунд (opds-viewer): React 19 + Vite 6 + react-cosmos; разметка LayoutExample.tsx зеркалит layout.xsl 1:1 (те же BEM-классы) и портируется руками; CSS собирается в dist/static/index.css и вручную копируется в static/style.css (727 vs 728 строк — дрейф уже есть); React-бандл в проде не используется вовсе; у main.js нет исходника ни в одном репо; фикстуры cosmos — выдуманные строки, реальные данные (кириллица, 200-элементные папки, отсутствующие обложки) впервые видны в проде.

Наблюдаемый дрейф: попап в JSX хардкодит «Fiction, Drama, Classic» там, где XSL рендерит реальные dc:*-поля; поиск есть в JSX, но <!-- search hidden --> в XSL; атрибуты data-element/data-focus-group, от которых зависит main.js, существуют только в XSL — плейграунд их не видит.

External Context. Помимо таймлайна Chrome: техника Джейка Арчибальда (2025) — HTML-namespaced <script> первым потомком XML-корня исполняется браузером и может заменить documentElement отрендеренным HTML; SaxonJS/XJSLT компилируют XSLT в JS; пиры (Kavita, Komga) разделяют OPDS-эндпоинт для читалок и человеческий UI; cosmos-export (уже в package.json) — паттерн «один исходник фикстур, два билд-выхода»; JSX↔XSLT-мосты в экосистеме бросают из-за импеданса.

1

Конвергенция: один TS-рендерер, HTML рендерится на этапе синка

markup-source-of-truth Confidence 90% Complexity High
Description
Целевая архитектура из четырёх ходов, каждый самостоятелен: (1) viewer переезжает в opds-generator как bun-workspace ui/; (2) пишется один зависимость-свободный TS-модуль renderFeed(xmlDoc) → html — вся логика layout.xsl (value-of, for-each, MIME→формат) выражается шаблонными литералами (~200 строк); (3) генератор вызывает его в существующем событийном каскаде и кладёт index.html рядом с feed.xml в каждую папку /data; разводка аудиторий не требует content-negotiation — читалки уже живут в графе feed.xml-ссылок (rel="subsection" в реальном data/feed.xml указывает на /comics/feed.xml и т.п.), браузеры переезжают на папочные URL (index index.html + корневой redirect — две строчки nginx); паттерн Kavita/Komga; (4) Vite hot-reload’ит тот же модуль в плейграунде на реальных фикстурах (идея 5). layout.xsl и LayoutExample.tsx удаляются — двух диалектов больше нет.
Basis
direct: folder-meta-sync.ts:127 — рендеринг фида уже per-folder-операция каскада, index.html — ещё один артефакт того же цикла; external: Astro/Eleventy — единый язык шаблонов для HTML и XML вместо моста между двумя языками; Kavita/Komga — разделение OPDS-протокола и человеческого UI; reasoned: вся логическая поверхность layout.xsl (прочитан целиком) — value-of, for-each, contains()-цепочка format-from-mime — тривиально выражается в JS, ни одна XSLT-фича не является несущей; direct: спайк 2026-07-14 на живом dev-стенде: черновой renderFeed() (~90 строк, fast-xml-parser — уже зависимость) отрендерил 8 index.html по всему /data; правка nginx — ровно 2 строки (index index.html + корневой redirect); вся роутинг-матрица зелёная с первого запуска: / и /comics/ отдают HTML с прод-CSS и папочной навигацией, feed.xml//opds/скачивание книг нетронуты (curl-проверка; сами фиды байт-в-байт не менялись, так что путь читалок не затронут по построению); визуально подтверждено пользователем.
Rationale
Единственное направление, которое решает обе проблемы пользователя и дедлайн Chrome одним механизмом: правка шаблона = один файл, hot-reload в дев-режиме, автоматическая доставка в прод при следующем синке. Сохраняет скрытую суперсилу XSLT — работу без JS (попапы уже на CSS-чекбоксах, main.js — только прогрессивное улучшение). Ошибки рендеринга всплывают в логах сервера при синке, а не в браузере пользователя.
Downsides
Самый большой объём работы из всех идей; +N файлов index.html в /data (место — копейки); nginx-механика снята спайком 2026-07-14 (2 строки, вся матрица зелёная — см. Basis); остаётся: человек со старой закладкой на feed.xml увидит сырой XML (опционально лечится Accept-redirect'ом); полнота рендера — спайк покрыл только карточки, попап/метаданные/breadcrumb-логика впереди, эквивалентность с XSL требует доказательства (закрывается идеей 3 — golden-харнессом); не проверялась ветка @check_initializing при пустом /data.

Mod-loader: Vite-прокси поверх живого docker с подменой /static/

real-data-dev-loop Confidence 85% Complexity Low-Med
Description
Как мод-лоадер в играх накладывает дев-каталог поверх файловой системы игры: Vite dev server проксирует живой docker (localhost:8080 — реальная библиотека, реальные фиды), но перехватывает /static/* и отдаёт layout.xsl / CSS / main.js из исходников с hot reload. Браузер открывает настоящий фид, а шаблон и стили — твои локальные, с мгновенной перезагрузкой.
Basis
direct: server.ts:53 — PI-href корне-относительный (/static/layout.xsl), значит кто отдаёт страницу, тот и контролирует, какой layout.xsl загрузится; верификатор подтвердил механическую осуществимость прокси-оверлея (same-origin под nginx-алиасом /static/).
Rationale
Убивает главную жалобу («разработка далеко от реальных данных») за ~30 строк vite.config, не дожидаясь большой миграции: дев-цикл «сохранил CSS — вся твоя реальная библиотека перерендерилась» в миллисекундах. Работает с сегодняшней XSLT-архитектурой и переживёт миграцию (оверлей поверх любых /static-артефактов).
Downsides
Требует запущенного docker; это дополнение к cosmos (изолированные состояния компонентов остаются в фикстурах), а не замена; XSL-правки перестанут hot-reload’иться после ухода от XSLT — но CSS/JS-оверлей останется актуальным.

Stop-the-bleed: sync-static + golden-харнесс как страховка миграции

artifact-pipeline Confidence 90% Complexity Low
Description
Terraform-принцип plan/apply для static/: (а) bun run sync-static — собирает viewer и пишет артефакты прямо в opds-generator/static/; sync:check падает при байтовом дрейфе (CI/pre-commit); (б) golden-харнесс — Playwright открывает N реальных URL из docker, снимает DOM-снапшоты и скриншоты, diff против эталонов. Пункт (б) — это доказательство эквивалентности «до/после» для любой миграции рендеринга (идеи 1 и 6).
Basis
direct: дрейф уже случился — static/style.css 727 строк vs dist/static/index.css 728 (подтверждено diff’ом); reasoned: в проекте нет ни одного теста отрендеренной страницы — тестовое дерево покрывает форматы/очередь/nginx, но не человеческий UI; при двух рукосинхронизируемых диалектах каждая правка шаблона не верифицирована.
Rationale
Самое дешёвое в списке (день работы), останавливает текущее кровотечение и — главное — превращает «страшную миграцию до 17 ноября» в механический проверяемый рефакторинг: переписал рендерер → харнесс доказал пиксельный паритет.
Downsides
Не устраняет причину (два диалекта) — только делает дрейф громким; golden-снапшоты требуют стабильной тестовой библиотеки книг (фиксированный набор в docker-compose.test).

main.js получает исходник, unpkg увольняется

artifact-pipeline Confidence 95% Complexity Low
Description
Перенести Gridnav + focus-trap-обвязку в viewer/src/gridnav.ts (вторая Vite-entry → static/main.js), подключить tabbable/focus-trap как npm-зависимости вместо рантайм-скриптов с unpkg, и добавить cosmos-фикстуру с реальными атрибутами data-element/data-focus-group — клавиатурная навигация впервые становится тестируемой до прода.
Basis
direct: layout.xsl:66-67 — <script src="https://unpkg.com/tabbable/..."> без пина версии: self-hosted приложение с жёсткой рантайм-зависимостью от аптайма unpkg; досье: main.js «не имеет исходника ни в одном репо — разрабатывался вслепую»; атрибуты, от которых он зависит, отсутствуют в DOM плейграунда.
Rationale
Самый интерактивный код продукта (2D-клавонавигация, фокус-ловушки) — единственный код вообще без дев-цикла. Идея направление-нейтральна: нужна при любом исходе (XSLT, Арчибальд, sync-time HTML) и закрывает реальную дыру надёжности — прод-UI сейчас ломается, если unpkg лёг.
Downsides
Практически нет; чуть больше размер main.js (бандлится tabbable/focus-trap — несколько КБ).

Кассеты: fixtures:pull собирает худшие реальные случаи из /data

real-data-dev-loop Confidence 85% Complexity Low
Description
VCR-паттерн из API-тестирования: скрипт обходит /data (обычное файловое зеркало — feed.xml/entry.xml на папку, подтверждено верификатором), выбирает патологические записи — самый длинный заголовок, отсутствующая обложка, глубочайшая вложенность, самая большая папка — и записывает их версионируемыми фикстурами cosmos. Фикстуры перестают быть выдумкой («Fiction, Drama, Classic») и становятся greatest-hits реальных краевых случаев.
Basis
direct: LayoutExample.tsx:147-148 — данные плейграунда буквально захардкожены; досье: «кириллица, длинные описания, отсутствующие обложки, 200-элементные папки — всё впервые видно в проде»; /data — walkable-файлы, не API.
Rationale
Дополняет идею 2: прокси даёт живую итерацию, кассеты — оффлайн-работу и закреплённые регрессионные случаи, переживающие изменения библиотеки. Десяток строк Bun. Эти же фикстуры кормят рендерер идеи 1 и харнесс идеи 3 — тройное использование.
Downsides
Снапшоты устаревают (лечится перезапуском скрипта); нужен доступ к /data или запущенный docker при harvest’е.

Самотрансформирующийся фид: script-bootstrap вместо PI (техника Арчибальда)

runtime-rendering Confidence 75% Complexity Medium
Description
Альтернативный ответ на дедлайн Chrome, сохраняющий фирменное свойство «один URL — и валидный OPDS, и человеческая страница»: вместо <?xml-stylesheet?> генератор инжектит HTML-namespaced <script> первым потомком корня фида; браузер исполняет его даже внутри XML-документа, скрипт читает XML-DOM, в котором живёт, и заменяет documentElement отрендеренным HTML. Скриптом может стать сегодня мёртвый билд viewer’а — dist/static/index.js впервые получает работу.
Basis
external: Jake Archibald, 2025 — работает с Content-Type: text/xml; direct: server.ts:53 — инжекция PI уже одна строка-шаблон, замена полезной нагрузки — минимальный диф; direct: смок-тест 2026-07-14 на живом dev-стенде: <script xmlns="http://www.w3.org/1999/xhtml" src="/static/bootstrap.js" defer="defer"/> первым потомком <feed> в data/test/feed.xml — Chromium отрендерил HTML (все 17 записей, ошибок нет), Moon+ Reader и FB Reader переварили фид со script-элементом: папка открывается, книга качается. Firefox/Safari визуально не подтверждены.
Rationale
Минимальный диф от текущей архитектуры среди всех пост-XSLT путей; объединяет рендереры плейграунда и прода в один JS-бандл. Честный соперник идеи 1 — выбор между ними и есть главное решение.
Downsides
Главный риск снят смок-тестом 2026-07-14 (см. Basis): реальные читалки пользователя (Moon+ Reader, FB Reader) толерантны к script-элементу; выборка — 2 читалки, экзотические парсеры не проверялись. Остаётся: теряется работа без JS — то, что XSLT давал бесплатно и что чекбоксовые попапы проекта уже ценят; рендер-баги всплывают в браузере пользователя, а не в логах синка; Firefox/Safari не подтверждены визуально.

Rejection Summary

# Идея Причина отклонения
1 «Нулевой билд»: удалить PostCSS, писать CSS прямо в static/ Опровергнуто верификацией: random() используется 14× в variations.css и запекает реальные oklch-значения (static/style.css:680-683) — пайплайн несёт нагрузку, его удаление = ручной подсчёт оттенков навсегда
2 Плейграунд рендерит настоящий layout.xsl через XSLTProcessor Инвестиция в умирающий шаблонный язык; XSLTProcessor нигде не используется (greenfield-обвязка); осмысленно только при «XSLT-максимализме», который сам отклонён
3 Скоупированный JSX→XSLT-компилятор Внешние свидетельства: мосты такого рода бросают на импедансе; строго доминируется единым TS-рендерером (идея 1)
4 Lit + Declarative Shadow DOM как общий слой компонентов Фреймворк ради 4 компонентов — непропорционально; plain-TS-рендерер даёт то же без новой зависимости и DSD-поддержки браузеров
5 XSLT-максимализм: layout.xsl как источник, XJSLT/SaxonJS-компиляция Тяжёлый тулчейн и незрелость компиляторов против ~200 строк шаблонных литералов; «ноль переписывания разметки» не окупает владение экзотическим рантаймом
6 DOM-parity-тест JSX vs XSL Chore, а не направление; как компонент влит в идею 3, устаревает с приходом единого рендерера
7 Публиковать cosmos-export как styleguide на прод-nginx Публичная поверхность для инструмента с единственным пользователем — соло-мейнтейнером; ту же страховку даёт golden-харнесс
8 Арчибальд-конвергенция (монорепо + рендерер + script-bootstrap) Доминируется идеей 1: жертвует работой без JS ради «одного URL», который sync-time HTML и так обеспечивает через content-negotiation
9 Монорепо как отдельная идея Влито в идею 1 как фаза 1 — самостоятельная ценность («один CI») недотягивает до направления
10 Правило «должно работать без JS» как фильтр Не идея, а критерий выбора — использован в обосновании ранжирования (идея 1 > идея 6)
