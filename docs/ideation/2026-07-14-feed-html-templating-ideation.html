Templating for src/render/feed-html.ts

Should the hand-rolled string renderer be replaced with a small js-to-html templating approach? And if so, which one, and what has to land first so the swap is safe?

  • date: 2026-07-14
  • topic: feed-html-templating
  • mode: repo-grounded
  • focus: use a small js-to-html library instead of hand-built string concatenation (suggested: Alpine.js, htmx, data-template, uhtml, TwinSpark, sprae)

Grounding Context

Codebase context. src/render/feed-html.ts is 175 lines: two escape helpers (escapeHtml, escapeAttr) and eight composition functions building HTML via template literals with 15+ manual escape call sites. HTML is rendered server-side at sync time: renderHtml(model) returns a string written to /data/<folder>/index.html; nginx serves it statically. Browser JS (static/main.js, 23.8 KB: gridnav + focus-trap) is progressive enhancement only; the book popup works with zero JS via CSS :target.

Hard constraints (still valid). The convergence plan merged earlier today (docs/plans/2026-07-14-001) fixed two decisions. KTD-1: one FeedModel, two serializers (renderXml splices verbatim fragments; renderHtml consumes parsed fields only). KTD-7: src/render/* imports no Bun/node APIs so the Vite playground can import the renderer directly. That plan also concluded "no templating library" — a migration-scoped restriction, per the author: now that the convergence has landed, a library is back on the table if its pros outweigh the cons. The constraints (string output, no DOM at render time, browser-importable) remain binding.

Verified findings from evidence scouts. escapeAttr escapes " but never ': a real, untested injection surface, since ebook metadata is untrusted input. model.title is escaped independently in two functions; card and popup duplicate cover/title/author/meta markup. The 11 unit tests + 1 e2e assert exact byte sequences (countOccurrences, indexOf ordering) that would break on harmless attribute reordering. tsconfig.json has jsx: "react-jsx" with no jsxImportSource. Nothing mechanically enforces KTD-7 today.

External research (2026). Of the six suggested libraries, only uhtml-ssr renders strings server-side. Alpine.js and sprae are client-side DOM-reactive runtimes; htmx and TwinSpark consume server HTML rather than produce it; data-template is a stale client-side DOM templater. The genuine string-output field converges on two poles: tagged template + auto-escape (hono/html, uhtml-ssr) and JSX-to-string (@kitajs/html, wrapped by Elysia's html plugin). Eta does not escape by default; vhtml is unmaintained; lit-html SSR is the wrong weight class. escape-html (74M weekly downloads) covers just the escaper.

Institutional learnings. No docs/solutions/ knowledge base exists. A render spike (docs/ideation/render-spike.ts) validated the hand-rolled approach in July 2026.

Topic Axes

  • template-composition — how markup is authored in feed-html.ts: escaping, readability, repetition
  • render-constraints — browser-importability, sync-time string output, dual-renderer FeedModel contract
  • client-enhancement — the progressive-enhancement JS layer (gridnav, popups) and where client-side libraries legitimately sit

Ranked Ideas

Jump to:
  1. Auto-escaping tagged template: hono/html, uhtml-ssr, or ~30 hand-rolled lines
  2. Harness before swap: hostile cassette + golden render gate + structural tests
  3. Component formalization: exported render units + pre-escaped view objects
  4. Make KTD-7 mechanical: browser-purity gate + playground spike protocol
  5. POST /rehtml: re-render every index.html from existing feed.xml in seconds
  6. Hyperscript node tree + single serializer (zero deps)
  7. JSX-to-string via @kitajs/html (spike-gated)

1. Auto-escaping tagged template: hono/html, uhtml-ssr, or ~30 hand-rolled lines

template-composition Confidence 85% Complexity: Low

Description: Replace escapeHtml/escapeAttr and every manual call site with an html`` tagged template that auto-escapes every interpolated value by default, plus raw() (or a branded SafeHtml return type) for composed fragments. Existing template literals keep their exact shape: the migration is "prefix the backtick, delete the escape calls." Three implementations of the same design, in ascending dependency appetite: a hand-rolled ~30-line tag (zero deps, keeps today's "no library" conclusion technically intact), uhtml-ssr (the one suggestion from the original list that genuinely renders strings server-side; same tagged-template ergonomics), or hono/html (actively maintained, audited escaping, isomorphic, a few hundred bytes).

Basis
direct: feed-html.ts:3-9: escapeAttr chains escapeHtml then replaces only ", never ' (verified by the adversarial pass); feed-html.ts:168 shows two escape calls in a single interpolated line, repeated across 8 functions. Evidence dossier: "No tests for … XSS attempts." external: hono/html and uhtml-ssr both implement exactly this contract (auto-escape + raw opt-out, pure string output, no DOM).
Rationale
Today's safety net is "did the author remember to call the right escaper here": 15+ opportunities to forget, and one already-latent gap (single quotes). Auto-escaping inverts the default: forgetting becomes safe, unescaped output becomes a visible raw() decision. This is the user's "kinda template" feel with the smallest possible diff, and it satisfies KTD-7 untouched (all three variants are pure string operations, browser-importable).
Downsides
Hand-rolled variant means owning the escaper (mitigated by idea 2's hostile cassette). Library variants add a dependency (knip/oxlint bookkeeping, supply-chain surface); the plan's no-library stance was migration-scoped and is now negotiable. None of the variants fixes javascript: href sanitization (see idea 2).

2. Harness before swap: hostile cassette + golden render gate + structural tests

migration-safety Confidence 90% Complexity: Medium

Description: Land a renderer-migration harness as its own PR before touching feed-html.ts, with four parts. (a) A hand-authored hostile.xml cassette: " onmouseover=alert(1) titles, javascript: acquisition hrefs, </script> breakouts, apostrophes, newlines, double-escape probes. (b) A render:check golden gate: commit renderHtml(parseFeed(cassette)) output for all cassettes as golden HTML, gate with git diff --exit-code, exactly mirroring the existing build:ui:check idiom. (c) Migrate the byte-exact test assertions (countOccurrences, indexOf ordering) to parse-then-query structural assertions using fast-xml-parser, already a dependency. (d) Name the zero-JS popup invariant explicitly: a test asserting no on*= attributes ever appear in rendered output.

Basis
direct: dossier: "No tests for: double-escaping scenarios / attribute value edge cases (quotes, apostrophes, newlines) / XSS attempts"; feed-html.test.ts asserts countOccurrences(html, 'class="card card--book"') === 5 and indexOf ordering (verified); both break under harmless attribute reordering; package.json:23 build:ui:check proves the repo already trusts the commit-and-diff gate idiom. reasoned: ebook metadata comes from downloaded internet files: attacker-controlled strings whose only barrier is this renderer's escaping; and javascript: href sanitization is a model-layer rule that no candidate library provides, so only this corpus would catch it.
Rationale
This is what makes every other idea on the list cheap and reversible. With the harness in place, any candidate (tagged template, hyperscript, JSX, uhtml-ssr) is validated in one test run against real cassettes plus an adversarial corpus, instead of hoping 25 substring checks covered enough. It also fixes something real regardless of direction: the apostrophe gap and the untested injection surface exist today.
Downsides
Golden files churn on intentional markup changes (that is also their point: reviewers see the actual HTML diff). Structural test migration is a test-only PR that must be reviewed carefully to preserve intent.

3. Component formalization: exported render units + pre-escaped view objects

template-composition Confidence 80% Complexity: Low

Description: Formalize the component boundaries that already exist implicitly. Export renderCover/renderMeta/renderDownloads (card and popup already share them; renderCover is called twice inside renderBook with different args). Add a toView(entry) mapper that escapes each field once into a plain object, so render functions read pre-escaped fields instead of re-escaping per usage site (model.title is currently escaped independently in renderHtml and renderHeader). Extract the multi-line ternaries (author line, summary line) into named fragment functions. The playground gains per-component preview panes since it can import the units directly.

Basis
direct: feed-html.ts:41/66: same field escaped in two functions; :116-117: cardCover/popupCover from one shared function; :136-139: a conditional spanning the template-literal boundary with hand-maintained \n + 16-space indentation, flagged by the dossier as the file's worst readability spot.
Rationale
This is the "do regardless" move: it reduces the repetition the original request reacts to, with zero dependencies and zero KTD friction. The resulting view-object + component shape is exactly what any templating library expects as input, so it converts a later library adoption from rewrite into rename. If the team is not ready to reverse this morning's decision, this still delivers most of the felt improvement.
Downsides
Doesn't change the escaping default (idea 1 does that); done alone, the manual-escape discipline remains, just centralized.

4. Make KTD-7 mechanical: browser-purity gate + playground spike protocol

render-constraints Confidence 85% Complexity: Low

Description: KTD-7 ("src/render/* imports no Bun/node APIs; stays Vite-importable") is currently enforced by nothing (verified: no script or CI step checks it). Add one: bun build src/render/*.ts --target=browser as a CI step (or an oxlint no-restricted-imports rule scoped to src/render) that fails on any Bun/node import, direct or transitive. Pair it with a standing protocol: every candidate library gets a 10-minute spike inside bun run dev:ui (Vite HMR) before serious consideration.

Basis
direct: KTD-7 quoted verbatim in the plan («Модули src/render/* не импортируют Bun- и node:fs-API… это держит их импортируемыми Vite-плейграундом») with zero enforcement anywhere in package.json (verified); ui/playground/main.ts:6-7,27 is the live import path a bad dependency would break. reasoned: SSR libraries are the classic source of transitive node:stream/node:buffer imports; the risk KTD-7 guards against goes up exactly when a library is adopted.
Rationale
Reframes today's decision entirely: "browser-importability" stops being an argument against libraries and becomes a 5-second acceptance test each library passes or fails. hono/html, uhtml-ssr, @kitajs/html each get an objective verdict instead of a debate. Cheapest item on the list; follows the repo's existing gate idiom.
Downsides
A transitive-import gate can produce false confidence about runtime behavior (a lib can be importable yet behave differently under Vite HMR; hence the paired spike protocol).

5. POST /rehtml: re-render every index.html from existing feed.xml in seconds

render-constraints Confidence 75% Complexity: Medium

Description: Template changes currently require a container restart or full POST /resync; the whole sync machine re-runs (metadata extraction, covers, queue cascade) just to change a CSS class in the markup. But renderHtml(parseFeed(feedXml)) is already the tested composition: a standalone pass can walk /data, read each existing feed.xml, and rewrite only index.html: no book files touched, no queue involved. Expose as POST /rehtml behind the same Basic Auth as /resync, or trigger automatically at boot when a renderer-version hash embedded in index.html doesn't match.

Basis
direct: test/unit/render/feed-html.test.ts:11-12 imports and composes parseFeed + renderHtml; this exact pipeline is the tested path (verified); KTD-1 confirms renderHtml consumes only parsed fields, and parse-feed.ts exists precisely to rebuild the model from feed.xml. reasoned: nothing in the composition touches book files or metadata extraction, so a re-render pass is architecturally independent of the sync cascade.
Rationale
Changes the cost model of every other idea here: template iteration drops from "full library resync" to seconds over existing XML. Whichever renderer direction wins, the swap itself and every future markup tweak becomes cheap to roll out and cheap to roll back.
Downsides
New endpoint = new surface: needs auth wiring (mirror /resync), nginx route, and a decision about concurrency with a running cascade (skip-if-syncing flag, same pattern as the existing isSyncing guard).

6. Hyperscript node tree + single serializer (zero deps)

template-composition Confidence 65% Complexity: Medium

Description: The structure-over-syntax alternative to idea 1: a ~40-line h(tag, attrs, ...children) building a plain node tree, plus one serialize() that owns escaping, attribute quoting, and void elements in a single choke point. Conditionals become plain TS (entry.author && h("p", {class:"popup__author"}, entry.author)) instead of ternaries spliced into literals; whitespace becomes the serializer's job, not the author's. 

h("a", { href: a.href, class: "popup__download-btn" },
  formatFromMime(a.type))
Basis
direct: feed-xml.ts:11 has feed.toXml({ prettyPrint: true }): the XML half of this renderer is already builder + emitter, while the HTML half hand-splices strings (verified structural asymmetry). reasoned: serializer discipline (protobuf, pandoc writers): output built as data and emitted by one encoder makes escaping bugs impossible by construction and lets tests assert on the tree rather than substring-match serialized bytes.
Rationale
The only option where FeedModel → output becomes mechanically inspectable (a tree you can walk) rather than a string you can grep. Restores symmetry with renderXml, keeps zero dependencies, and makes structural testing (idea 2c) nearly free.
Downsides
Markup stops looking like HTML: a real taste cost versus idea 1, and a bigger diff (every literal rewritten as calls). Competing head-on with ideas 1 and 7 for the same slot; the team should pick exactly one pole.

7. JSX-to-string via @kitajs/html (spike-gated)

render-constraints Confidence 55% Complexity: Medium–High

Description: The high-ceiling pole: rename to feed-html.tsx, set jsxImportSource to @kitajs/html, and author <BookCard entry={entry}/> with typed props. JSX renders directly to strings (no VDOM, no DOM), auto-escapes by default, and, unlike template literals, is visible to the formatter, the linter, and the type checker: a clas= typo becomes an error, and oxfmt can finally reflow the markup.

Spike first: two unverified blockers

tsconfig.json currently sets jsx: "react-jsx" with no jsxImportSource (verified; it defaults to React), and one config must drive both Bun's native transpiler (docker prod + tests) and Vite's esbuild (playground). Historical Bun issue #3528 (non-React jsxImportSource mis-transpilation) is from 2023 and may be fixed; its current status is unverified. Half-day spike: render one cassette through both toolchains and diff the outputs.

Basis
direct: tsconfig.json sets "jsx": "react-jsx" with no jsxImportSource key (verified by the adversarial pass). external: @kitajs/html renders JSX to strings, auto-escapes, fastest-in-class in its own benchmarks; Bun transpiles JSX natively from tsconfig; Elysia's html plugin wraps it (ecosystem validation).
Rationale
The biggest authoring-ergonomics win on the table and the only option that scales if the viewer grows (search pages, tag pages, sorting views): real components, typed props, tool-visible markup. The cost is also the largest: shared build config touching the playground, plus the spike burden.
Downsides
Highest integration risk of any candidate; reverses this morning's KTD most visibly; JSX idioms invite onClick-shaped habits that the zero-JS popup contract forbids (idea 2d's invariant is the guard).

Rejection Summary

44 raw ideas from five framing agents were merged into 20 clusters; an independent adversarial pass verified quotes against the repo (no fabrications found) and issued verdicts. Cut, with reasons:

# Idea Reason Rejected
1 Native <dialog> popup (delete focus-trap/tabbable, ~20 KB) Verifier: weak. CSS :target doesn't drive dialog modal state; this redesigns the zero-JS contract the tests enforce, not a rider on the renderer swap. Worth its own spike later.
2 JSON-island popup dedup (render card once, hydrate popup) Breaks the zero-JS popup guarantee asserted by existing tests; the trade was admitted but never resolved. Real duplication problem; revisit only if the zero-JS requirement is consciously relaxed.
3 Client-rendered shell (one static shell.html + parseFeed→renderHtml in browser) Re-litigates KTD-1 merged the same day with no new evidence; the playground precedent doesn't transfer: production serves no-JS clients and crawlers the playground doesn't. Reopen only as an explicit user decision.
4 sprae behavior-attributes enhancement layer Weakest grounding of the near-survivors: the claimed conflict with the dialog idea was asserted, not demonstrated; needs a concrete before/after to be discussable.
5 Page shell as a real .html file with slots The Bun-side ?raw-import story is unverified (Vite side works today); reintroduces the dual-toolchain seam KTD-7 exists to avoid.
6 escape-html micro-swap Subsumed: it is the minimal branch of idea 1.
7 XSLT-at-build-time (feed.xml as the literal template) Self-rejected boundary stake: violates KTD-7 (native dep, breaks the playground path). Its surviving insight is idea 5.
8 Freshness-gate hardening (hash-based build:ui:check) Real (commit d745c9f already bit this) but orthogonal to the templating decision; backlog item.
9 Alpine.js, htmx, TwinSpark, data-template (from the original suggestion list) Category error for this problem: all four are client-side runtimes that consume or bind to server HTML; none produces server-side strings. Their legitimate home (enhancement layer) produced no surviving candidate this round.
10 CSS-class coherence gate (single source of class names) Sound but crowded out at the 7-survivor cap; partially covered by idea 2's golden diffs, which make class drift visible in review.
— axis: client-enhancement Deliberate gap: no standalone survivor. All four candidates (dialog, JSON island, sprae, client shell) either break or must spike the zero-JS contract; the zero-JS invariant itself survives inside idea 2.
