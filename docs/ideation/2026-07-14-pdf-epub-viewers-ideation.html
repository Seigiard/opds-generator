Ideation

In-browser PDF & EPUB viewers

Date 2026-07-14 Topic pdf-epub-viewers Mode repo-grounded

Focus: add PDF and EPUB viewers for files; possibly a single HTML file with all data baked in (pdf.js or epub-js); show a view button near the "download" button.

48 raw ideas 5 frames × agents 7 survivors 5/5 axes covered

Grounding Context

Codebase Context. OPDS catalog generator (Bun + nginx, Docker). /data mirrors /books; each book gets entry.xml, cover.jpg, thumb.jpg, and a symlink to the source file. One FeedModel per folder renders feed.xml (OPDS reader apps) and index.html (browsers) at sync time. Browser popups are hash + CSS :target and work with no JS; main.js is progressive enhancement. static/ artifacts are generated by build:ui, committed, and freshness-gated in CI. No CSP headers today; no viewer/reader code exists; nginx serves book files with Content-Disposition: attachment (pinned by an e2e test) and implicitly supports range requests (sendfile on).

Past learnings. The single-model-two-renderers pattern is deliberate and extensible; renderers stay pure TS (no Bun/node imports) so they run in the Bun handler, the Vite playground, and browser tests; every new generated file in /data needs dual watcher safety (data-adapter classification + watcher.sh regex).

External research (high value). Kavita is the reference comp: its Nov 2025 reader overhaul + annotation system signals users want progress/annotations, not just page-turning. epub.js (futurepress) is effectively unmaintained; intity/epub-js is an active fork but needs webpack; foliate-js is the standout for self-hosted no-CDN setups (native ESM, zero build, MIT, EPUB/MOBI/AZW3/FB2/CBZ + experimental PDF adapter, powers Foliate in production, unstable API, pin via submodule). pdf.js: build on the display layer (pdf.mjs + worker + cmaps/), not Mozilla's viewer.html. Baking book bytes into HTML: base64 inflates ~33-37%, 2MB data-URI ceilings exist, and SingleFile deliberately avoids base64-ing binaries. Inline the JS library, never the book.

1

One shared reader shell, powered by foliate-js

viewer-page-generation Confidence 80% Complexity Medium
Description
Ship exactly one static reader page (built by build:ui, committed to static/ like main.js), opened full-page as /static/read.html#/sci-fi/Dune.epub. One runtime: foliate-js (vendored as a pinned git submodule; native ESM, so no webpack enters the repo) covers EPUB/MOBI/AZW3/FB2/CBZ and PDF through its pdf.js adapter — validated by hand on a complex PDF in the foliate demo. pdf.js remains a vendored dependency of the adapter (pdf.mjs + worker + cmaps/, its own hashed asset directory), lazy-loaded only when a PDF opens. The engines are inlined/vendored (the repo already inlines focus-trap + tabbable); the book's bytes are never baked in; they're fetched from the existing acquisition URL. The "View" button is a plain anchor added inside renderDownloads(), which already has href and MIME type in scope. Iterate on the shell in the Vite playground against real cassette books, same loop the HTML renderer already uses.
Axis
viewer-page-generation (absorbs runtime-bundling decisions)
Basis
direct: src/render/feed-html.ts:165-174: renderDownloads() maps entry.acquisitions with a.href and formatFromMime(a.type) already in scope; ui/scripts/build-ui.ts: single Bun.build pass with focus-trap/tabbable inlined (the vendoring precedent). external: foliate-js: native ESM, zero build step, MIT, EPUB/MOBI/AZW3/FB2/CBZ + experimental PDF, powers Foliate in production; epub.js unmaintained; pdf.js display layer is Mozilla's stable public API. direct: base64-baking rejected on evidence: 33-37% inflation, 2MB data-URI ceilings, SingleFile precedent (web research digest). external: foliate-js README: the zip.js loader supports HTTP range requests (EPUBs can stream partially from nginx with no server-side unpacking), and the library ships progress.js, search.js, overlayer.js (annotations), and an opds.js OPDS 1.x feed parser the shell can point at feed.xml directly.
Rationale
This is the direct build path for the user's ask, with three deliberate corrections to the opening sketch: one shared page instead of a generated file per book (zero new watcher-safety surface, zero cascade changes; every per-book artifact historically required dual watcher-safety work); foliate-js instead of the unmaintained epub.js (8 of 13 catalog formats from one dependency, PDF included, vs 2 formats from two libraries); and "single file" applied to the reader chrome only, never the book bytes (preserves range-request streaming for large scans). The full-page surface (not the metadata popup) is deliberate: the popup's CSS and focus trap are tuned for a short-lived modal, not sustained reading.
Downsides
foliate-js has an explicitly unstable API (submodule pinning is mandatory, upgrades are manual diffs); the PDF adapter is "proof-of-concept, highly experimental" in the authors' own words, so it needs a playground smoke test on real catalog PDFs pinned into CI, with the direct pdf.js display layer as the cheap fallback if a regression lands; pdf.js's worker + cmaps can't ride the current single-bundle pattern, forcing the repo's first hashed-immutable asset directory (a new caching pattern next to the deliberately no-cache style.css/main.js). The contrasted fork (per-book viewer.html generated by a third renderer in the cascade) buys deep self-containment (cp -r a book folder and it reads) at the cost of a per-artifact watcher-safety surface; it lost on cost, not on merit.
2

CSP + scripting-disabled EPUB as the gating prerequisite

viewer-runtime-bundling Confidence 90% Complexity Low
Description
Before any EPUB rendering ships, add a strict Content-Security-Policy header in nginx scoped to the viewer surface (script-src 'self' + worker allowances for pdf.js), configure the reader with EPUB scripting disabled, and confine book HTML to a sandboxed iframe. Treat it like email clients treat sender HTML: hostile by default.
Axis
viewer-runtime-bundling
Basis
external: foliate-js's authors flag scripted EPUB content as fundamentally unsafe to sandbox (WebKit bug 218086 defeats iframe sandboxing) and mandate strict CSP blocking all non-'self' scripts. direct: the repo ships zero CSP headers today (nginx.conf.template), and the viewer would run same-origin with the Basic-Auth-protected /resync endpoint.
Rationale
An EPUB is a zip of arbitrary HTML+JS. Rendering untrusted books same-origin without CSP turns every downloaded ebook into a potential XSS vector against the host, including credentialed endpoints. This is the one decision that is catastrophically expensive to retrofit and costs about two nginx lines to do up front. The verifier ranked it the most important gate in the set: it should precede, not follow, ideas 1 and 5.
Downsides
CSP must be scoped carefully so it doesn't break the existing no-CSP catalog pages; pdf.js needs worker-src (and version-dependent wasm) allowances, so the policy needs one e2e test of its own to avoid silent viewer breakage on upgrades.
3

A format capability matrix decides where "View" appears

view-entry-ui Confidence 85% Complexity Low
Description
Jellyfin doesn't have one playback strategy; it has a direct-play/transcode matrix per item. Do the same: a VIEWABLE_FORMATS registry co-located with MIME_TYPES in src/types.ts maps each of the 13 extensions to a viewing strategy: client engine (foliate/pdf.js formats), server-derived (idea 5's page images for DJVU/CBR), or download-only. renderDownloads() consults it, so "View" appears only where it actually works.
Axis
view-entry-ui
Basis
direct: src/types.ts:16-31: the MIME table is already the single registry nginx and format handlers both consume, and formatFromMime() (feed-html.ts:17-30) already branches per MIME at render time. external: Jellyfin/Plex direct-play-vs-transcode matrices are the canonical answer to "heterogeneous formats, one client surface"; foliate-js covers neither DJVU nor CBR.
Rationale
The classic failure mode here is shipping "a viewer" that means "an EPUB viewer": a blanket View button that renders a blank pane for djvu/fb2 users is a worse first impression than today's forced download. A one-registry-line-per-format matrix makes coverage a designed, reviewed property. When idea 5 later adds a server-derived path for DJVU, flipping its row turns the button on with zero render-logic changes.
Downsides
The matrix is honest but visible: users of download-only formats can see their books are second-class. Keeping the matrix truthful requires updating it in the same PR as any engine change: cheap, but a convention to enforce.
4

Byte delivery is already done: verify it, don't build it

byte-delivery-nginx Confidence 85% Complexity Low
Description
Three facts collapse the assumed nginx workstream to near-zero. (1) Content-Disposition: attachment only governs navigation/save-as; fetch() and XHR receive the body regardless, so pdf.js/foliate-js can consume the existing acquisition URLs untouched, and the e2e-pinned attachment contract survives. (2) Range requests are implicitly on (sendfile on), but that's an inference, so add one e2e case asserting 206 Partial Content for a Range: request against a real book before betting streaming on it; the same mechanism seeds OPDS-PSE comic page-streaming later. (3) The only thing the header actually blocks is the browser's native PDF viewer via navigation; an optional ?disposition=inline nginx map (S3's pattern) restores that as a zero-JS day-1 rung.
Axis
byte-delivery-nginx
Basis
reasoned: RFC 6266 scopes Content-Disposition to user-agent presentation/save handling on navigation; no browser applies it to programmatic fetches; verified as a platform fact by the fresh-context reviewer. direct: nginx.conf.template:72-81 (the unconditional attachment header), test/e2e/nginx.test.ts:74-83 (pins the header on GET but never exercises fetch or Range). external: S3's response-content-disposition query param: canonical prior art for caller-chosen disposition on the same object.
Rationale
Multiple ideation frames independently proposed nginx header surgery to "unblock" the viewer; the verification pass showed that entire workstream solves a non-problem for JS viewers. Stating this explicitly changes every estimate: the viewer's byte-delivery layer is finished before it starts, pending one cheap 206 check. Also flagged: RATE_LIMIT_MB throttles viewer fetches identically to downloads (a deliberate check, not a surprise).
Downsides
If the 206 check fails on the book location block, large scanned PDFs must fully download before first paint; that outcome would reshape idea 1's PDF story, which is exactly why the check precedes library commitment.
5

A zero-JS reading tier: server-derived chapters and page images

reading-experience-state Confidence 60% Complexity High
Description
Apply the project's progressive-enhancement identity to reading itself. At sync time, format handlers derive readable artifacts the way they already derive covers: sanitized chapter HTML for text formats (softened by a later finding: foliate's zip.js loader can range-stream EPUBs client-side, so server-side unzip is now the zero-JS tier's story, not the only streaming story), and pre-rendered page-image pyramids (IIIF Level-0 pattern) for DJVU/CBR/scanned PDFs using the ddjvu/pdftoppm tools already in the Docker image. The base tier is plain HTML with prev/next anchors: readable on e-ink browsers (Kindle/Kobo, arguably this catalog's most natural browser audience), over 2G, and under full CSP lockdown. Idea 1's JS engines become the enhancement layer on top, exactly as main.js enhances the popup today.
Axis
reading-experience-state
Basis
direct: the repo's documented interaction contract: popups work with no JS, main.js is progressive enhancement (CLAUDE.md); src/formats/pdf.ts:94 and src/formats/djvu.ts:101 invoke pdftoppm/ddjvu per book today. Corrected per verification: those calls are hardcoded to page 1 for covers; the CLI tools support arbitrary page ranges, but page-range rendering is new code, not a flag flip. reasoned: server-side extraction also converts the EPUB security problem (idea 2) from "sandbox untrusted JS in the client" to "sanitize once on the server", which is strictly safer.
Rationale
This is the only path that reaches the format tail (DJVU has no browser library at all) and the only one that serves JS-incapable reading devices. It reframes the roadmap question from "which JS library" to "who reads in a browser", and it makes the viewer an extension of the architecture's core bet (pre-compute at sync, serve statically) rather than a SPA bolted onto a static site.
Downsides
The big-ticket item: disk cost (page images for scanned books are large), slower syncs, new per-book artifacts requiring the dual watcher-safety dance, and real new code in every format handler. Honest sequencing: idea 6 is this idea's cheap first increment; full chapter/page derivation only makes sense if previews prove demand.
6

Quick-Look previews as the shippable v1

reading-experience-state Confidence 70% Complexity Medium
Description
Break the assumption that "view" must mean the whole book. A catalog's browsing job is "is this the book I want?", answered by a bounded preview: first chapter of an EPUB/FB2, first ~5 pages of a PDF/DJVU as images. Add an optional getPreview() to FormatHandler (mirroring macOS Quick Look's index-time generators), write the artifact next to entry.xml, and surface a "Preview" affordance in the existing :target popup via sandboxed iframe. Because previews are kilobytes by construction, every delivery concern (data-URI ceilings, range tuning, rate limits) disappears, and one page-range rendering primitive serves both this and idea 5's full pyramids later.
Axis
reading-experience-state
Basis
direct: src/formats/types.ts:14-17: FormatHandler is exactly getMetadata() + getCover(); the per-book derived-artifact mechanism (cover extraction, cascade write site in book-sync.ts:105-106) is proven. Corrected per verification: current pdftoppm/ddjvu calls are page-1-only cover code; page-range extraction is new work in each handler, not "changing one flag." external: macOS Quick Look's decoupling of preview generation from display.
Rationale
Cheapest genuine increment that tests user appetite for in-browser reading before any reader architecture locks in, ships value for all 13 formats (including the ones no JS library reads), and de-risks the roadmap: if previews go unused, the full reader was the wrong bet; if they're loved, their artifacts are reusable substrate for idea 5.
Downsides
It's not the reader the user asked for; it must be framed as a sequencing choice, not a substitute. New per-book artifacts need the watcher-safety additions; preview extraction adds sync time per book.
7

Reading progress in localStorage, keyed by stable entry id

reading-experience-state Confidence 65% Complexity Medium
Description
Podcast apps solved position-sync under this exact constraint: the server is dumb static files, so position lives client-side and the list wears the state. The reader shell writes {position, percent, updatedAt} to localStorage keyed by the entry's atom id; reopening resumes; main.js (as pure enhancement) badges catalog cards with progress and a "continue reading" affordance. No server state, no new artifacts, degrades to nothing without JS.
Axis
reading-experience-state
Basis
direct: FeedEntry.id exists (feed-model.ts:16, populated from the source atom <id>) but, per a verification finding, the popup anchors actually use positional book-${index} (feed-html.ts:114). Named prerequisite: re-key the hash scheme from index to stable id (and confirm id stability across resyncs) before building on it. external: Kavita's 2025 reader overhaul shipped progress/annotations as the headline, the community signal for what separates a demo from a reader.
Rationale
A viewer without resume is a demo; with resume it's a reader people return to. This is the smallest credible slice of the feature the reference comp spent years on, fully reversible, and its arrival is the right moment to introduce the project's first client-side persistent state, a precedent worth a deliberate yes rather than an accidental one.
Downsides
Per-browser only (honest labeling needed: no cross-device sync); the hash-rekey prerequisite touches deep-link semantics that existing users may have bookmarked; localStorage is invisible to the server, so "recently read" can never appear in feeds without a philosophy change.
Sequencing note

A natural ladder emerged across frames: 4 (verify 206, optional zero-JS inline-PDF rung) → 2 (CSP gate) → 3 (matrix) + 1 (reader shell) → 7 (progress) → 6/5 (previews, then the zero-JS tier if demand proves out). Ideas 6 and 5 can also lead if "preview first, reader later" is the preferred risk posture.

Rejection Summary

Idea Reason rejected
View-first hierarchy (cover click opens reader, download demoted) Scope overrun: reorders the catalog's primary action, a product-identity change beyond "add a view button"; backed only by comp positioning, no repo evidence.
Portable shelf (/data as copy-out offline library, file:// support) Scope overrun as framed: a distribution feature, not a viewing feature; demoted to a passive constraint (prefer relative URLs in new artifacts).
Standalone ?inline nginx header variant to "unblock" JS viewers Premise refuted: fetch() ignores Content-Disposition, so JS viewers need no header change; surviving zero-JS native-PDF value folded into idea 4.
Per-book viewer.html via third renderer in the cascade Costlier fork of idea 1: every per-book artifact adds watcher-safety surface and re-render-on-upgrade cost; kept as the named contrast inside idea 1.
Reader as a new #read-book-N state of the existing popup Interaction-shape mismatch: popup CSS (overflow:none) and focus trap are tuned for a short-lived metadata modal, not sustained reading; full-page surface won inside idea 1.
"Reader shell" mega-bundle (shared page + matrix + hash links + progress as one package) Two of four components carried unstated gaps (popup-as-reader mismatch; hash-rekey prerequisite); superseded by ideas 1, 3, 7 with prerequisites named.
Single HTML file with book bytes baked in (user's opening sketch, literal form) Refuted on evidence: base64 inflates 33-37%, 2MB data-URI ceilings, kills range streaming; the self-contained instinct survives as "inline the reader JS, never the book" inside idea 1.
epub-js (intity fork) as the EPUB engine Duplicates a stronger option: requires JSZip + webpack (a bundler the repo deliberately avoids) and covers 1 format where foliate-js covers 7; futurepress original is unmaintained.
Hashed immutable per-engine asset dirs; playground as viewer test bench Not rejected; absorbed into idea 1 as implementation-level decisions (asset layout, dev loop).
PDF-dedicated pdf.js display-layer lane (separate from foliate) Dropped after hands-on validation: foliate's pdf.js adapter handled a complex PDF in the demo; single-runtime bet taken, direct pdf.js display layer kept as the named fallback inside idea 1.
